# Notion運用ルール（商品企画スキル共通）

## 保存先
- **企画マスターページ**: プロジェクトDBの正式テンプレ「【新商品】企画段階定型」を複製して作る（`notion-template.md` 参照）。1企画＝プロジェクトDBの1ページ。企画概要（1〜6）とマーケ調査（1〜13）はこのページに埋める
- **ドシエ**（内部の作業メモ）: 企画マスターページの「6.販売価格」直下にサブページとして格納する
- フィードバックログは従来どおり田代メモ (115b9edd0d8280d4abb2e0b6fbfee593) 直下に置く
- **テンプレの型・欄と各スキル出力の対応・作成手順は `notion-template.md` を単一仕様書とする（このファイルと必ずセットで読む）**

## 分割書き込みルール（タイムアウト防止）
- 長文の書き込みは1回で行わず、ページ作成 → 追記系の更新で3〜5回に分割する
- 目安: 1回の書き込みは3セクション程度まで

## 企画ドシエ（企画ごとの単一情報源）
商品企画の状態は、チャットの要約ではなく**企画ごとに1つのドシエ（Notionサブページ）に蓄積する**。後工程はアンカー定義や証拠を必ずドシエから読み直すこと。チャット上の要約を信用しない（工程を跨ぐたびに前提が劣化する伝言ゲームを防ぐ）。

- **作成タイミング**: ジャッジメントテストに進む案が決まった時点（product-planning Step 2の末尾。idea-judgment 単体実行時にドシエが無ければ、その場で作成）。この時点で企画マスターページ（テンプレ複製）も作り、ドシエはその「6.販売価格」直下の子ページとして作る（`notion-template.md` の作成手順に従う）
- **場所**: 企画マスターページ（プロジェクトDB）の「6.販売価格」直下の子ページ
- **ドシエのセクション構成**（内部の作業メモ。マスターページの企画概要／マーケ調査欄とは別物）:
  1. **アンカー定義**: 誰の悩みか・建前・本音・裏付け証拠（レビュー引用等）
  2. **フック・FV・納得の理屈**: product-idea の最終出力
  3. **ジャッジメントテスト記録**: 軸×判定×確信度、攻防の要点、戦略確認事項、ユーザーoverrideとその理由
  4. **検証課題**: グレー軸・持ち越し宿題（解消されたら更新する）
- **マスターページへの転記**: ドシエの「プロジェクト概要」相当は企画概要（1〜6）へ、マーケ調査結果はマーケ調査トグル（1〜13）へ転記する（対応は `notion-template.md`）
- **運用ルール**: 各スキルは開始時にドシエ（＋マスターページ）を読み、終了時に自分の担当セクション（ドシエ／マスターページ）を追記・更新する（分割書き込みルール適用）

## 商品企画フィードバックログ（スキルを跨いだ学習の仕組み）
- ページ名「商品企画フィードバックログ」（田代メモ直下。存在しなければ最初の書き込み時に作成）
- 1判定＝1行の箇条書きで追記する:

  `[YYYY-MM-DD] テーマ | 案名 | フック(30字) | ステージ | 判定 | 理由`

  - ステージ: アイデア選定 / ジャッジメントテスト / マーケ調査 / 発売後
  - 判定: 採用 / 見送り / 没 / ヒット / 不発
  - 理由: ユーザーの言葉をなるべくそのまま残す（**最重要フィールド**）

### 読み込みルール（product-idea / idea-judgment / market-research が実行前に行う）
1. ログの直近30件を読む
2. 「没・見送りの理由」からNGパターンを抽出（例: ギミック感が強い、ターゲットが狭い、類似品が既にある）
3. 「採用・ヒット」から勝ちパターンを抽出
4. 出力の冒頭で「今回避けるNGパターン」を1〜2行で宣言し、生成・判定条件に反映する
5. ログが存在しない・空の場合はスキップしてよい

### 書き込みルール（判定が発生した全スキルが行う）
- ユーザーの判定（採用/見送り/没）が出た直後に1行追記する
- 見送り・没の理由が不明なら一言だけ確認してから記録する（しつこく聞かない。もらえなければ「理由未記録」と書く）
- AIの判定とユーザーの判定が食い違った場合（AIはGoだがユーザーは見送り等）は、**ユーザーの判定と理由**を記録する — これが次回精度を上げる最重要データ
